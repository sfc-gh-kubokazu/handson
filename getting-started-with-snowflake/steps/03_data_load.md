# STEP 3: データのロード

## このセクションでやること

S3 上の Citi Bike データを Snowflake に取り込みます。

- **実行ファイル**: `../scripts/02_data_load.sql`

---

## Snowflakeの概念: データロードの仕組み

### COPY INTO コマンド
ステージ上のファイルをテーブルに一括コピーするコマンドです。

```sql
COPY INTO <テーブル名>
FROM @<ステージ名>
FILE_FORMAT = <ファイルフォーマット名>
PATTERN = '.*csv.*';  -- ファイル名パターン（省略可）
```

**冪等性**: 一度ロード済みのファイルは再実行しても自動的にスキップされます。  
誤って 2 回実行してもデータが重複しません。

### ウェアハウスサイジング
データロード時はウェアハウスを大きくすることで高速化できます。

| サイズ | vCPU 相当 | クレジット/時間 |
|---|---|---|
| XS | 1 | 1 |
| S | 2 | 2 |
| M | 4 | 4 |
| **L** | **8** | **8** |
| XL | 16 | 16 |

- ウェアハウスのサイズ変更はその場で即時反映されます
- ロード完了後は必ず元のサイズに戻しましょう（コスト管理）

### ウェアハウスの分離（ベストプラクティス）
同じウェアハウスをデータロードとアナリティクスクエリで共用すると、  
お互いのリソースを食い合ってパフォーマンスが落ちます。  
→ **用途別にウェアハウスを分けること**が推奨です。

```
compute_wh      ← データロード用（Small）
analytics_wh    ← アナリティクスクエリ用（Large）
```

1. `../scripts/02_data_load.sql` を実行

### 実行後の確認ポイント

**データがロードされた:**
```sql
SELECT COUNT(*) FROM citibike.public.trips;
-- 6,000,000 件前後が表示される
```

**COPY 結果の詳細を確認:**
```sql
SELECT *
FROM TABLE(information_schema.copy_history(
    TABLE_NAME => 'TRIPS',
    START_TIME => DATEADD(HOUR, -1, CURRENT_TIMESTAMP())
));
```

**アナリティクス用ウェアハウスが作成された:**
```sql
SHOW WAREHOUSES;  -- analytics_wh が表示される
```

## 補足: Snowsight UI機能（データプレビュー / データリネージ）

ロードしたデータを SQL を書かずに確認・追跡できる UI 機能を紹介します。  
`Catalog » Database Explorer` から `CITIBIKE » PUBLIC » TRIPS` を選択すると、上部にタブが並びます。

### データプレビュー
**「データプレビュー」タブ** で、テーブルの**先頭最大100行**を SQL なしで即座に閲覧できます。

- ロードが正しく行われたかを目視で確認できる
- 列のデータ型に応じて、ソート（昇順/降順）・小数精度の増減・桁区切り表示・パーセンテージ表示などの調整が可能
- **「テーブル詳細」タブ** では 種別・所有ロール・作成日時・行数・サイズ・SQL定義(DDL)、**「列」タブ** では列名・型の一覧を確認できる

> 注意: プレビューはクエリを実行するためウェアハウスを消費します（ただし100行なのでコストは僅少）。

### データ品質（Data Quality）
**「データ品質」タブ**（プレビュー機能）では、データの健全性を継続的に検証できます。「ロードしたデータは本当に正しいか？」を自動・定期的にチェックする仕組みで、機能自体（DMF）は GA 済みです。タブは2つの機能で構成されます。

#### ① Data Profile（プロファイル）— セットアップ不要
テーブル/ビューの統計を**自動収集**して表示します。設定ゼロで使えます。

- 行数 / 最終更新時刻
- 列ごとの **NULL数・最小/最大・最頻値・値の分布**
- → 構造・内容・潜在的な品質課題を素早く把握

#### ② Monitoring（モニタリング）— DMFによる継続チェック
オブジェクトに紐付けた **DMF（Data Metric Function）の結果・スケジュール・トレンド**を表示します。失敗したチェックをドリルダウンし、影響資産や違反レコードまで調査できます。

**中核となる3概念:**

| 概念 | 説明 |
|---|---|
| **DMF** | データの属性を測る関数（NULL数、最終更新からの経過時間 等）。値を返すだけで良し悪しは判定しない＝部品 |
| **Expectation（期待値）** | DMFの戻り値と比較する基準。`DMF + 期待値 = データ品質チェック`。基準外の値は「違反」として報告 |
| **Anomaly Detection（異常検知）** | 過去データから予測範囲を学習し外れ値を自動検知（現在は volume / freshness 対応） |

**主なシステムDMF（設定不要）:**

| 品質ディメンション | DMF |
|---|---|
| Accuracy（正確性） | `NULL_COUNT` / `NULL_PERCENT` / `BLANK_COUNT` / `BLANK_PERCENT` |
| Uniqueness（一意性） | `DUPLICATE_COUNT` |
| Volume（量） | `ROW_COUNT` |
| Freshness（鮮度） | `FRESHNESS`（最終更新からの経過秒数。TIMESTAMP列が対象） |
| 統計 | `AVG` / `MIN` / `MAX` |

→ システムDMFに無い指標は**カスタムDMF**を自作できます。

#### DMFの実行タイミング
DMFの実行タイミングは**テーブル/ビュー単位**で `DATA_METRIC_SCHEDULE` を設定します（そのテーブル上の全DMFに適用）。方式は3つ。

| 方式 | 設定例 | 内容 |
|---|---|---|
| **①インターバル** | `SET DATA_METRIC_SCHEDULE = '5 MINUTE'` | 指定間隔で定期実行。指定値は **5 / 15 / 30 / 60 / 720 / 1440分**（最短5分・最長1日）。**デフォルト60分** |
| **②CRON** | `USING CRON <式> <タイムゾーン>` | cron式で柔軟に指定。最短間隔5分 |
| **③トリガー** | `'TRIGGER_ON_CHANGES'` | **DML（INSERT/DELETE等）で変更された時のみ**実行。通常/一時/トランジェント/外部/Iceberg/Dynamicテーブルで可（**ビューは不可**、リクラスタリングでは発火しない） |

- **アドホック実行**: スケジュールとは別に、DMFをクエリ内で直接呼べばその場で測定も可能
- **一時停止**: `SET DATA_METRIC_SCHEDULE = ''`（空文字）で全DMFをサスペンド。`UNSET` でデフォルト60分に戻る
- **コスト観点**: トリガーベースは「変更時のみ」なので、更新頻度が低いテーブルではサーバーレスクレジットを節約できる

```sql
-- 例: TRIPS に鮮度チェックを設定（5分間隔で監視）
ALTER TABLE citibike.public.trips SET DATA_METRIC_SCHEDULE = '5 MINUTE';
ALTER TABLE citibike.public.trips
  ADD DATA METRIC FUNCTION SNOWFLAKE.CORE.NULL_COUNT ON (starttime);
```

#### 嬉しいポイント
- **「正しいと言えるデータ」を仕組みで担保**（データオーナーシップに直結）
- **Cortex Data Quality**: メタデータと利用パターンからAIがチェックを自動提案（手動定義不要、データはSnowflake内に留まる）
- **結果は中央のEvent Tableに集約**（OpenTelemetry標準形式）。`DATA_QUALITY_MONITORING_RESULTS` ビュー/関数でクエリでき、ダッシュボード化・アラート連携が容易

> ロード直後は「このデータは正しいか？」を確認するのが第一歩。**プレビュー（中身を見る） → プロファイル（品質を理解する） → DMF（継続監視する）** の順で見せると、データオーナーシップの訴求に綺麗に繋がります。

### データリネージ（系列）
**「系列」タブ** で、データがどのオブジェクトからどのオブジェクトへ流れたか（系譜）をグラフで可視化できます。

- 上流（ソース）/下流（ターゲット）を `+ / -` で1階層ずつ展開
- オブジェクト間の矢印をクリックすると、**その関係を生成した SQL** が表示される
- 列レベルのリネージも確認可能（インパクト分析に有用）

#### リネージが「表示される」起因
リネージは **実際にデータが移動した操作から自動・リアルタイムに生成** されます。主な起因は以下の通りです。

| 起因となる操作 | リネージ表示 | 補足 |
|---|---|---|
| `CREATE TABLE ... AS SELECT`（CTAS） | ◯ | ソーステーブル → 新テーブル |
| `INSERT` / `MERGE` | ◯ | データが移動した分が記録される |
| `CREATE VIEW` / マテリアライズドビュー | ◯ | ビュー依存（複数テーブルのJOINも） |
| `COPY INTO`（ステージ → テーブル） | ◯ | **ステージもノードとして表示される** |
| **Snowpipe**（自動取り込み） | ◯ | 内部は COPY なので、ステージ → テーブルが出る |
| **Snowpark**（`save_as_table` 等の書き込み） | ◯ | 内部で SQL(CTAS/INSERT/MERGE) に変換されるため |
| Dynamic Table | ◯ | グラフには出るが、自身の「系列」タブは持たない |

#### リネージが「表示されない」ケース
| 操作・状況 | 理由 |
|---|---|
| ストアドプロシージャ経由のデータ移動 | 追跡されない（`GET_LINEAGE` も対象外） |
| 分断されたクエリ（変数に入れて別クエリで INSERT 等） | データの連続性が追えない |
| JOIN / フィルタにのみ使い、データが流れていないテーブル | リネージに含まれない |
| 一時テーブル / 共有DB / INFORMATION_SCHEMA のオブジェクト | 対象外 |

> **このハンズオンの TRIPS は、ロードしたばかりで上に何も構築していない「単独のソーステーブル」なので、系列タブはほぼ空になります。**  
> これは正常な挙動で、「リネージ＝複数オブジェクト間のデータの流れがあって初めて意味を持つ」ことを表しています。  
> 実際のパイプライン（ソース → 加工ビュー → 分析/AI層）でどう見えるかは、講師の画面で多段のリネージ例をお見せします。

#### Snowflake 外のデータと繋げたい場合
dbt や Apache Airflow など外部ツールの系譜も含めたい場合は、**External Lineage**（OpenLineage 連携）を使うと、外部ノードとして同じグラフに統合表示できます（※列レベル非対応・Snowsight UIでのみ確認可能）。

## よくある質問

**Q: `COPY executed with 0 files processed.` と表示される**  
A: すでにロード済みのファイルはスキップされます。  
テーブルを `TRUNCATE TABLE trips;` してから再実行するとロードできます。

**Q: ウェアハウスがサスペンドしていてロードが遅い**  
A: `AUTO_RESUME = TRUE` なので初回アクセス時は起動に数秒かかります。正常です。
